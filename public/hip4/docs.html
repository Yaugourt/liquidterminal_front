What is HIP-4?

HIP-4 is Hyperliquid's prediction market system. It lets anyone bet on the outcome of real-world events — "Will BTC be above $68k tomorrow?", "Who will win the 100m dash?" — using HYPE tokens.

There is no official documentation for HIP-4. Everything in this document was reverse-engineered from the on-chain bytecode, real transaction analysis, and the HyperCore API. This is field work — not theory.

On HyperEVM testnet we track two deployments: legacy V1 (bytecode + reconstructed ABI) and V2 (verified Solidity + compiled ABI). Claims, Merkle leaves, and fee accounting differ between versions — see Full ABI — V1 vs V2 for the side-by-side table and JSON ABIs (HIP4Contest.v1.abi / HIP4Contest.v2.abi next to these pages).

1.
An admin creates a contest (a question with multiple sides/outcomes) and sets the entry fee.
2.
Anyone deposits HYPE to pick a side. You can also trade the outcome tokens on the HyperCore L1 order book.
3.
When the event resolves, the admin posts a Merkle root on-chain. Winners submit a proof and claim their HYPE.
Platform fee on V1 is explicit: platformFeeBps() returns 90 — 0.9% skimmed from the published reward pool, immutable on-chain. On V2 there is no bps getter; the fee is implicit from pool totals vs payouts, with owner withdrawPlatformFee instead of legacy sweepUnclaimed.
The two layers — HyperCore vs HyperEVM

HIP-4 lives across two separate systems that talk to each other. This is the key thing to understand:

HyperCore L1
  • Native order book — this is where you trade
  • Outcome tokens are traded as spot assets
  • Price = implied probability (0.00 to 1.00)
  • No EVM interaction needed to buy/sell
  • Block numbers in the ~526M+ range
HyperEVM (Chain 998)
  • EVM-compatible chain — this is where the contract lives
  • Holds the prize pool in HYPE
  • Admin publishes Merkle roots here
  • Winners claim rewards here
  • Block numbers in the ~48M range
The bridge between them is at address 0x2222…2222 — it holds ~998M HYPE. When you send HYPE to it on the EVM side, the funds appear on HyperCore L1 (and vice versa). It's a system-level contract, not a user-deployed one.
How outcome tokens are mapped to the order book

Each outcome token on HyperCore L1 has an asset index. The formula we discovered:

asset_idx = 100_000_000 + parseInt( outcomeId + sideId )

The outcomeId and sideId are concatenated as strings, not added as numbers.

CoinOutcomeSideString concatAsset idxMeaning
#90 90 "9" + "0" = "90" 100000090 Hypurr wins the 100m dash
#91 91 "9" + "1" = "91" 100000091 Usain Bolt wins the 100m dash
#19520 19520 "1952" + "0" = "19520" 100019520 BTC > $68,147 — Yes
#19521 19521 "1952" + "1" = "19521" 100019521 BTC > $68,147 — No
How we confirmed this: A real user transaction used a: 100000090 to trade coin #90. The formula holds for every observed asset.
Code — Connect to the contract

All examples below use ethers.js v6 and target the V1 deployment unless noted. The RPC is public — no API key needed. For V2, swap address and use HIP4Contest.v2.abi.

import { JsonRpcProvider, Contract } from "ethers"; // HyperEVM Testnet const RPC = "https://rpc.hyperliquid-testnet.xyz/evm"; const ADDR_V1 = "0x4fd772e5708da2a7f097f51b3127e515a72744bd"; const ADDR_V2 = "0x6d86b21e853758f5719408633e6bcb2cfd50cf07"; // V1: reconstructed ABI (not verified — reverse-engineered) // On deployed V1, selector 0x71c5ecb1 is the public getter merkleRoots(uint256) — not getMerkleRoot(). const ABI = [ "function owner() view returns (address)", "function platformFeeBps() view returns (uint256)", "function getContestPool(uint256 contestId) view returns (uint256)", "function merkleRoots(uint256 contestId) view returns (bytes32)", "function deposit(uint256 contestId, uint256 sideId) payable", "function createContest(uint256 contestId, uint256 entryFee)", "function publishMerkleRoot(uint256 contestId, bytes32 root, uint256 rewardPool)", "function finalizeContest(uint256 contestId)", "function refund(uint256 contestId, uint256 sideId, address user)", "function sweepUnclaimed(uint256 contestId)", "function transferOwnership(address newOwner)", "event DepositReceived(uint256 indexed contestId, uint256 indexed sideId, address depositor, uint256 amount)", "event Claimed(uint256 indexed contestId, uint256 indexed sideId, address claimer, uint256 amount)", "event ContestFinalized(uint256 indexed contestId)", "event MerkleRootPublished(uint256 indexed contestId, bytes32 root, uint256 rewardPool)", "event ContestCreated(uint256 indexed contestId, uint256 entryFee)", ]; const provider = new JsonRpcProvider(RPC); const contract = new Contract(ADDR_V1, ABI, provider);
Code — Read contract state
Get contest pool balance
const contestId = 595; const poolWei = await contract.getContestPool(contestId); const poolHype = Number(poolWei) / 1e18; console.log(`Contest #${contestId} pool: ${poolHype} HYPE`); // → Contest #595 pool: 26.64 HYPE
Check if a contest exists
// getContestPool returns 0 for non-existent contests const pool = await contract.getContestPool(999); const exists = pool > 0n; console.log(exists); // false — contest 999 doesn't exist
Read the Merkle root (check if contest is finalized)
const root = await contract.merkleRoots(595); console.log(root); // → 0xc8ed36d4... (finalized — root exists) // → 0x00000000... (not finalized — root is zero)
Get owner and fee
const owner = await contract.owner(); const feeBps = await contract.platformFeeBps(); console.log(`Owner: ${owner}`); // 0xe21c78037329d06fe0d6fefc4221aaa67cb0d135 console.log(`Fee: ${feeBps} bps`); // 90 bps = 0.9%

V2: there is no platformFeeBps() — use pool views and the verified ABI. See Full ABI — V1 vs V2.

Code — Listen to events
Watch deposits in real time
contract.on("DepositReceived", (contestId, sideId, depositor, amount) => { console.log(`New deposit on contest #${contestId}`); console.log(` Side: ${sideId}`); console.log(` From: ${depositor}`); console.log(` Amount: ${Number(amount) / 1e18} HYPE`); });
Scan past deposits (batch query)
const filter = contract.filters.DepositReceived(); const fromBlock = 48_000_000; const toBlock = 48_200_000; const logs = await contract.queryFilter(filter, fromBlock, toBlock); for (const log of logs) { const { contestId, sideId, depositor, amount } = log.args; console.log(`#${contestId} side=${sideId} ${depositor} ${Number(amount)/1e18} HYPE`); } // ⚠ Testnet RPC is slow — use small block ranges (50k–100k max)
Watch for contest finalization
contract.on("ContestFinalized", (contestId) => { console.log(`Contest #${contestId} has been finalized!`); }); contract.on("MerkleRootPublished", (contestId, root, rewardPool) => { console.log(`Merkle root published for contest #${contestId}`); console.log(` Root: ${root}`); console.log(` Reward pool: ${Number(rewardPool) / 1e18} HYPE`); });
Code — Deposit into a contest
Testnet only. This sends real transactions. The entry fee must match exactly — the contract reverts with "Incorrect entry fee" otherwise.
import { Wallet, parseEther } from "ethers"; const signer = new Wallet(PRIVATE_KEY, provider); const hip4 = new Contract(ADDR_V1, ABI, signer); const contestId = 595; const sideId = 0; // 0 = first outcome const entryFee = parseEther("0.1"); // must match the contest entry fee exactly const tx = await hip4.deposit(contestId, sideId, { value: entryFee }); const receipt = await tx.wait(); console.log(`Deposited! TX: ${receipt.hash}`); // Common errors: // "Incorrect entry fee" → wrong msg.value // "Already deposited for this round" → 1 deposit max per address // "Deposit deadline passed" → deposits are closed // "Contest does not exist" → invalid contestId

Code — Scan for active contests

There's no function to list all contests. You have to iterate and check pool balances.

// Brute-force scan — check contest IDs one by one async function findActiveContests(from, to) { const active = []; for (let id = from; id <= to; id++) { const pool = await contract.getContestPool(id); if (pool > 0n) { const root = await contract.merkleRoots(id); const finalized = root !== "0x" + "0".repeat(64); active.push({ id, pool: Number(pool) / 1e18, finalized, merkleRoot: finalized ? root : null }); console.log(`#${id}: ${Number(pool)/1e18} HYPE ${finalized ? "FINALIZED" : "OPEN"}`); } } return active; } // Scan IDs 1 to 700 const contests = await findActiveContests(1, 700); // Known active: #595 (26.64 HYPE), #596 (26.88), #604 (28.88), #608 (30.40) // IDs 1–594 return 0 — they either never existed or were swept
Code — Raw RPC calls (no library needed)

If you can't use ethers.js, here's how to query the contract with raw JSON-RPC using just fetch.

Read owner (slot 0)
// V1 contract address const res = await fetch("https://rpc.hyperliquid-testnet.xyz/evm", { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ jsonrpc: "2.0", id: 1, method: "eth_getStorageAt", params: [ "0x4fd772e5708da2a7f097f51b3127e515a72744bd", "0x0", // slot 0 = owner "latest" ] }) }); const { result } = await res.json(); const owner = "0x" + result.slice(26); // strip 24 zero chars // → 0xe21c78037329d06fe0d6fefc4221aaa67cb0d135
Call getContestPool via eth_call (V1)
// selector: 0x2a658bdf + contestId padded to 32 bytes · `to` = ADDR_V1 const contestId = 595; const data = "0x2a658bdf" + contestId.toString(16).padStart(64, "0"); const res = await fetch("https://rpc.hyperliquid-testnet.xyz/evm", { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ jsonrpc: "2.0", id: 1, method: "eth_call", params: [{ to: "0x4fd772e5708da2a7f097f51b3127e515a72744bd", data }, "latest"] }) }); const { result } = await res.json(); const poolWei = BigInt(result); console.log(Number(poolWei) / 1e18, "HYPE"); // 26.64

Known limitations & gotchas
  • !
    No explorer. There is no block explorer for HyperEVM testnet. You can't look up transactions on a website — everything must be done via RPC.
  • !
    V1 vs V2 provenance. V1 is not verified on-chain; names in this doc were reconstructed from bytecode and calldata. V2 matches verified Solidity in-repo and the compiled HIP4Contest.v2.abi file.
  • i
    Selector 0xb2447e34. On V1 bytecode this owner-only (uint256,uint256) call had no cracked name. On V2 the same selector is withdrawPlatformFee(uint256,uint256) (replaces legacy sweepUnclaimed). See Full ABI — V1 vs V2.
  • ~
    Slow RPC. The testnet RPC is rate-limited and slow. Event scans over large block ranges will timeout. Keep ranges under 100k blocks.
  • ~
    Claim surface. On V2, claim is 0x5d4df3bf with (contestId, index, recipient, amount, proof[]) and per-index bitmap accounting. On V1 the public claim(contestId, sideId, proof[]) shape is inferred from events/reverts; a reliable V1 selector was historically hard to pin from live calldata alone — prefer V2 ABI for new integrations.
  • ~
    Testnet only. Neither V1 nor V2 HIP-4 contest deployments are live on Hyperliquid mainnet (chain 999). Treat EVM addresses here as testnet-only.
  • i
    More than 2 sides. We observed sideIds up to 12 in real transactions. Contests can have many outcomes, not just binary Yes/No.
  • i
    Ownership is permanent. renounceOwnership() always reverts with "Renounce disabled". The owner cannot be removed — only transferred.
