What is this?

This hub documents HIP-4 on HyperEVM — the first HIP that runs contest logic in EVM smart contracts (create, deposit, Merkle settlement, claims). We now track two testnet deployments: legacy V1 (bytecode reverse-engineered) and V2 (verified source + ABI). Coverage includes ABIs, events, reverts, storage, real txs, bridge mechanics, live market scan, and JS/Python examples.

HIP-4 is Hyperliquid's parimutuel prediction stack: users back sides with native HYPE on the EVM contract while outcomes trade on HyperCore L1. Hyperliquid still publishes no official HIP-4 spec — this site is community field work, indexed and cross-checked on-chain.

How HIP-4 works — the short version
1. Create
The admin creates a contest — a question with multiple outcomes (sides). Each side becomes a tradeable token on HyperCore L1.
2. Trade & deposit
Users pick a side by depositing HYPE into the EVM contract, or by trading outcome tokens on the L1 order book in USDH.
3. Settle & claim
The event resolves. Admin posts a Merkle root on the EVM. Winners claim with proofs — V1 side-based model vs V2 per-index leaves + bitmap; platform fee is explicit on V1 (platformFeeBps) and implicit on V2 (pool accounting).
Theory — Why trade in USDH but rewards in HYPE? Unconfirmed

This is an unconfirmed theory based on our analysis. Hyperliquid has not explained this design choice.

What we observe
L1
Outcome tokens (#90, #19520, etc.) are traded on the HyperCore L1 order book with USDH as the quote currency — exactly like any spot market (BTC/USDH, ETH/USDH).
EVM
The prize pool, deposits, and claim payouts are all in HYPE — the native gas token of HyperEVM. The deposit() function is payable and uses msg.value.
Our theory on why
  • 1
    Architecture constraint. HyperCore L1 uses USDH as its native quote currency for all spot markets. The order book doesn't know about HYPE as a settlement token — it just prices assets in USDH. That's how the L1 works for every market, HIP-4 is no exception.
  • 2
    EVM native simplicity. On HyperEVM, HYPE is the native gas token (like ETH on Ethereum). Using msg.value for deposits is the simplest possible pattern — no ERC20 approve/transferFrom needed, no token contract dependency, less gas, fewer attack vectors.
  • 3
    Incentive alignment. By paying rewards in HYPE, winners are automatically exposed to the success of the Hyperliquid ecosystem. This creates organic buy pressure on HYPE and aligns participant incentives with the protocol's growth.
  • 4
    Layer separation. HyperCore (L1) and HyperEVM are two distinct systems connected by the bridge at 0x2222…2222. Each layer uses its own native currency. The L1 side handles price discovery in USDH; the EVM side handles settlement in HYPE. The bridge converts between them when needed.
Bottom line: USDH for trading (stable unit, native to L1 order book), HYPE for settlement (native to EVM, aligns incentives). Two layers, two currencies, one bridge connecting them. This is a pattern choice, not a limitation — it's how the dual-layer architecture naturally works.
Methodology

How this information was gathered — for transparency and reproducibility.

Bytecode analysis
Disassembled the contract bytecode. Extracted function selectors (PUSH4), revert strings, event topic0s, storage slot accesses (SLOAD/SSTORE), and CALLER checks.
Selectors & source
V1: keccak256 brute-force + calldata decode (publishMerkleRoot, contestExists, etc.). V1 owner-only 0xb2447e34 was unnamed until V2 ABI named it withdrawPlatformFee. V2: full interface from verified Solidity + bytecode selector check.
Transaction scan
Scanned blocks 47,900,000–48,200,000 via RPC. Decoded calldata and event logs from real transactions (deposits, createContest, finalizeContest, publishMerkleRoot).
HyperCore API
Queried the {type: "hip4"} endpoint for market metadata: outcomes, questions, side specs, coin prices. Cross-referenced with on-chain data.
Storage probing
Read storage slots 0–19 via eth_getStorageAt. Identified owner (slot 0), initialized flag (slot 2). Mappings use dynamic keccak256-derived slots.
State reads
Eth_call across contest IDs (pools, merkle roots, platformFeeBps on V1, contestRewardPools / getters on V2) to map live contests and published roots.
